## Cookie の動作確認

### 1. index.js で`document.cookie` プロパティを `console.log`で表示する

結果: 空文字列が表示される

理由: サーバが設定している Cookie には `HttpOnly` 属性が付与されており、JavaScript からアクセスできないため。

### 2. ブラウザの開発者コンソールで http://localhost:3000/ の Cookie を表示する

結果: `sid`という名前の Cookie が表示される

理由: ブラウザの開発者ツールは`HttpOnly` 属性が付与された Cookie も含めて、すべての Cookie を表示できるため。

### 3. ToDo アプリのタブをリロードする

結果: リロード前に作成した ToDo リストの内容が維持される

理由: ページをリロードしても、ブラウザは既存の Cookie（セッション ID）をサーバに送信する。サーバはこのセッション ID を用いてセッションごとに保存されているタスクデータを識別し、同じデータを返すため。

### 4. 同一ブラウザの異なるタブやウィンドウで http://localhost:3000/ を開いて ToDo リストの状態を確認する

結果: 既存のタブと同じ ToDo リストの内容が表示される

理由: 同一ブラウザ内では Cookie が共有されるため、新しいタブやウィンドウでも同じセッション ID が送信される。サーバは同じセッション ID を受け取り同じタスクデータを返すため。

### 5. シークレットウィンドウや異なるブラウザで http://localhost:3000/ を開いて ToDo リストの状態を確認する

結果: 空の TODO リスト（or異なる内容）が表示される

理由: シークレットウィンドウや異なるブラウザでは、通常のブラウザセッションとは別の Cookie ストレージが使用される。よって既存のセッション ID は共有されずサーバは新しいセッション ID を生成するため。

### 6. http://127.0.0.1:3000/ を開いて ToDo リストの状態を確認する

結果: 空の ToDo リスト（or異なる内容）が表示される

理由: Cookie はオリジン（スキーム、ホスト、ポート）に基づいて管理される。`localhost` と `127.0.0.1` は、どちらもローカルマシンを指すが、ブラウザからは異なるホストとして扱われる。そのため、`localhost:3000` で設定された Cookie は `127.0.0.1:3000` には送信されず、サーバは新しいセッション ID を生成するため。
